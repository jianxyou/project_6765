DocMerger

Adaptive Hierarchical Patch Merging for
Multi-Vector Visual Document Retrieval

Project Update — April 2026

Problem

  • Multi-vector VDR represents each page as hundreds of patch embeddings
  • State-of-the-art retrieval quality, but prohibitive storage
  • DocPruner (Yan et al., 2025): adaptive pruning via EOS attention thresholding
  • Works well at moderate compression, but degrades sharply at >75%
  • Key insight: pruning permanently discards information — can we do better?

DocMerger: Tri-Level Partitioning

Given importance scores I(dj) with mean μ and std σ:

Ppreserve = { dj | I(dj) > μ + k₁·σ } → keep as-is

Pmerge = { dj | μ − k₂·σ < I(dj) ≤ μ + k₁·σ } → cluster + weighted merge

Pdiscard = { dj | I(dj) ≤ μ − k₂·σ } → remove

  • Merging uses agglomerative clustering (cosine distance)
  • Centroids weighted by EOS attention importance
  • Merging is worse than keeping original, but better than discarding

Explored: Adaptive Hybrid (Direction 1)

  • Idea: Route each document to DocPruner or DocMerger based on attention entropy
  • High entropy (diffuse attention) → DocMerger
  • Low entropy (concentrated attention) → DocPruner

❌ Negative Result

  • Entropy doesn't reliably predict which method is better per document
  • Adaptive (0.5877) falls between DocPruner (0.5868) and DocMerger (0.5897) at ~70%
  • Conclusion: routing signal is too coarse — needs query-aware features

Explored: Learned Sparse Projection (Direction 3)

  • Idea: Replace clustering with a learned linear layer, trained via MaxSim distillation
  • Same-dataset: 0.6180 nDCG@5 at 80% compression (baseline: 0.6205)

⚠️ Does Not Generalize

Key Takeaways

  1. DocMerger outperforms DocPruner at >75% compression — consistent across 3 evaluation settings and 5 datasets
  2. Merging is worse than keeping originals, but better than discarding — the tradeoff pays off at high compression
  3. Adaptive per-document routing via entropy doesn't work — the signal is too coarse
  4. Learned projection shows high ceiling (0.62 at 80%) but doesn't transfer — needs query-agnostic training
  5. Practical recommendation: DocPruner ≤70%, DocMerger >75%
